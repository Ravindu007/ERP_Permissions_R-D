Enterprise Permission Architecture

1. Role-Based Access Control (RBAC)

RBAC replaces hardcoded role strings with a permission-based mindset. You define what a user can do as an action:resource string (e.g., delete:comment). This allows you to update permissions for a whole role in one place.

Note: Using join tables for many-to-many relationships ensures a user can have multiple roles, which is a senior developer best practice.

RBAC Logic Implementation

// In-memory representation of database results
const ROLES = {
  admin: ["view:comments", "create:comments", "update:comments", "delete:comments"],
  moderator: ["view:comments", "update:comments"],
  user: ["view:comments"]
};

function hasPermission(user, permission) {
  // Handles multiple roles per user
  return user.roles.some(role => ROLES[role].includes(permission));
}

2. Attribute-Based Access Control (ABAC)

ABAC is the most flexible system. It evaluates four components: **Subject**, **Action**, **Resource**, and **Environment/Attributes**. It handles "fine-grained" logic that RBAC cannot, like ownership or status-based locks.

The "Permissions" are not just strings; they are functions that compare the attributes of the Subject (User) and the Resource (Data).

ABAC Logic Implementation

const ABAC_RULES = {
  user: {
    todos: {
      // Logic: View if the author hasn't blocked you
      view: (user, todo) => !todo.author.blockedBy.includes(user.id),
      
      // Logic: Delete if completed AND user is owner or invited
      delete: (user, todo) => 
        todo.completed && (todo.userId === user.id || todo.invited.includes(user.id))
    }
  }
};

function checkPermission(user, resource, action, data) {
  return user.roles.some(roleName => {
    const rule = ABAC_RULES[roleName]?.[resource]?.[action];
    if (rule == null) return false;
    if (typeof rule === "boolean") return rule;
    return rule(user, data);
  });
}

3. Permission Handling in ERPs

In ERPNext, permission management is structured into a hierarchical three-level approach to ensure that users only access information relevant to their job profiles.

1
Role-Based Permissions Roles → DocType + Action Rights (read, create, submit, delete…)
▼ narrows to
2
Record-Based Permissions User Permissions → Specific Record Filtering
▼ narrows to
3
Field-Level Permissions Perm Levels → Individual Field Visibility & Editability

Level 1 Role-Based Permissions

At the highest level, permissions are governed by the roles assigned to a user.

  • Role Assignment: Every employee is assigned roles (e.g., Sales User, Sales Manager) based on their job profile. For recurring role combinations, you can create a Role Profile to automatically assign a set of roles to a new user at once.
  • Role Permissions Manager: This tool defines exactly what a role can do with a specific DocType (like a Customer or Sales Order).
  • Action Rights: Permissions can be fine-tuned to allow or disallow specific actions: read, create, edit, submit, delete, cancel, amend, share, print, import, or export.

Level 2 Record-Based Permissions

Even if a user has a role that allows them to see a certain DocType, you may want to restrict which specific records they can see.

  • User Permissions: Restricts access based on specific values. For example, a Sales User may have permission to see the Customer DocType, but a User Permission can limit them to seeing only the records of the specific customers they manage.
  • Advanced Controls: Further refine access, such as restricting a user to only see sales transactions from specific hotels or clients.

Level 3 Field-Level Permissions

The most granular level, where users are restricted from viewing or editing certain fields within a single document.

  • Perm Levels: Fields within a DocType are assigned a "Perm Level" (e.g., Level 0 for general fields, Level 1 for sensitive fields like Total Order Value or Ratings).
  • Access Control: Role Permissions can then be configured so that one role (like a Delivery Associate) only has read access to Level 0, while another (like an Account Manager) has read and write access to Level 1.

Additional Permission Tools

  • Role Permission for Pages and Reports: Restricts access to sensitive financial information such as Balance Sheets, ensuring only specific roles like CXOs or Directors can view them.
  • The Share Button: For temporary or unique scenarios, users can grant another user temporary read or write access to a specific document without permanently changing their role permissions.
Note on Previous Sections: While the system above is specific to ERPNext, the RBAC and ABAC patterns described earlier represent the standard architectural foundations used when building custom applications from scratch. RBAC aligns closely with ERPNext’s Level 1, while ABAC logic is often used programmatically to achieve the fine-grained control seen in Level 2 and Level 3.
